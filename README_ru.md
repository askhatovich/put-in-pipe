# Put-In-Pipe

Сервер потоковой передачи файлов с сохранением приватности участников. Сервер выступает в роли посредника между отправителем и одним или несколькими получателями, передавая файлы блоками без сохранения файла на диск целиком. Прямые соединения между клиентами не устанавливаются; IP-адреса участников не раскрываются другим сторонам.

## Возможности

- **Передача через сервер-посредник** -- клиенты обмениваются данными исключительно через сервер; прямые соединения между клиентами отсутствуют.
- **Потоковая передача блоками** -- размер передаваемого файла не ограничен дисковым пространством или памятью сервера.
- **Несколько получателей** в одной сессии, с настраиваемым верхним пределом.
- **Защита капчей** от автоматизированных злоупотреблений.
- **Сквозное шифрование** -- XChaCha20-Poly1305 на основе libsodium; ключ шифрования передаётся в фрагменте URL (`#`) и не передаётся на сервер.
- **REST API и интерфейс WebSocket** для получения событий передачи в режиме реального времени.
- **Встроенный веб-интерфейс** -- одностраничное приложение на Svelte 5, собираемое в единый HTML-файл и встраиваемое в исполняемый файл сервера.
- **Настройка** посредством файла формата INI.

## Сборка

### Только серверная часть

Требуется компилятор с поддержкой C++20 и CMake версии 3.16 или выше. Клиентская часть должна быть собрана до сборки серверной (см. ниже).

```sh
cmake -B build src
cmake --build build
```

### Полная сборка (клиентская и серверная части)

Дополнительно к компилятору C++ требуются Node.js версии 18 или выше и npm.

```sh
# 1. Сборка клиентской части
cd web
npm install
npm run build    # формирует web/dist/index.html (одностраничное приложение)
./embed.sh       # генерирует src/generated_index_html.h
cd ..

# 2. Сборка серверной части (с встроенным веб-интерфейсом)
cmake -B build src
cmake --build build
```

После завершения сборки запрос `GET /` возвращает полное веб-приложение; отдельный сервер статических файлов не требуется.

### Разработка клиентской части

Исходный код клиентской части расположен в каталоге `web/` и представляет собой стандартный проект Vite + Svelte 5.

```sh
cd web
npm install
npm run dev      # сервер разработки Vite с автоматической перезагрузкой (http://localhost:5173)
npm run build    # сборка для развёртывания → web/dist/index.html
```

При разработке следует запустить `npm run dev` и настроить браузер на проксирование API-запросов к серверной части (`http://localhost:2233`), либо параллельно запустить серверную часть и открыть адрес сервера разработки Vite.

**Используемые технологии:**
- [Svelte 5](https://svelte.dev/) -- реактивный фреймворк пользовательского интерфейса.
- [Vite](https://vite.dev/) совместно с [vite-plugin-singlefile](https://github.com/nickreese/vite-plugin-singlefile) -- сборщик, объединяющий все ресурсы в один HTML-файл.
- [libsodium-wrappers-sumo](https://github.com/nickvergessen/libsodium.js) -- реализация шифрования XChaCha20-Poly1305 в браузере.

**Структура проекта:**
```
web/
  src/
    App.svelte              -- корневой компонент и конечный автомат состояний приложения
    lib/
      api.js                -- клиент REST API
      ws.js                 -- клиент WebSocket с диспетчеризацией событий
      crypto.js             -- обёртка над libsodium для шифрования и расшифровки
      i18n.js               -- переводы (английский и русский) с автоматическим определением языка
      url.js                -- разбор и формирование фрагмента URL (#)
    components/
      EntryScreen.svelte    -- начальный экран (выбор режима: отправка или получение)
      Captcha.svelte        -- отображение и ввод капчи
      SessionSender.svelte  -- интерфейс отправителя
      SessionReceiver.svelte-- интерфейс получателя
      TransferComplete.svelte-- экран результата передачи
      MemberList.svelte     -- панель участников сессии
      ProgressPanel.svelte  -- индикатор прогресса передачи и состояния буфера
      Footer.svelte         -- статистика сервера и сведения об авторских правах
```

### Встраивание клиентской части в исполняемый файл

Сценарий `web/embed.sh` преобразует собранный файл `web/dist/index.html` в заголовочный файл C++ (`src/generated_index_html.h`), содержащий HTML в виде массива байтов. Серверная часть подключает этот заголовочный файл и возвращает его содержимое по запросу `/`.

Указанный заголовочный файл формируется автоматически и включён в `.gitignore`; его необходимо повторно генерировать после любого изменения клиентской части:

```sh
cd web && npm run build && ./embed.sh
```

## Запуск

```sh
# Показать справку
./build/put-in-pipe --help

# Сгенерировать конфигурационный файл по умолчанию
./build/put-in-pipe --generate-config ./config.ini

# Запустить сервер
./build/put-in-pipe --config ./config.ini
```

### Параметры командной строки

| Параметр | Описание |
|---|---|
| `-c, --config <путь>` | Путь к файлу конфигурации (по умолчанию: `/etc/put-in-pipe/config.ini`) |
| `-g, --generate-config <путь>` | Сгенерировать конфигурацию по умолчанию по указанному пути |
| `-h, --help` | Показать справку |

## Конфигурация

Полный перечень доступных параметров приведён в файле [`contrib/config.ini.example`](contrib/config.ini.example). Основные секции:

- **`[server]`** -- адрес и порт прослушивания.
- **`[client]`** -- максимальное число клиентов, пороговое значение для активации капчи, время жизни капчи, тайм-аут клиента.
- **`[session]`** -- максимальное число сессий, максимальный размер блока, размер очереди буфера, время жизни сессии, максимальное число получателей, длительность начальной задержки.
