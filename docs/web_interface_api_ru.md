## Web interface API

Документация Put-In-Pipe для фронтенда.

Put-In-Pipe — приложение для передачи файлов с сохранением приватности, где сервер выступает промежуточным буфером между отправителем и несколькими получателями. Сервер не хранит файл целиком (и, следовательно, не ограничивает размер передаваемого файла размером своего диска), а передаёт данные чанками: принял часть от отправителя, отдал получателям — и так до полной передачи. Клиенты никогда не соединяются напрямую — IP-адреса участников не раскрываются.

Архитектура построена на REST API и WebSocket. Клиент может быть как веб-приложением на JS, так и десктопным приложением на любом другом языке.

## Сквозное шифрование

Все передаваемые данные шифруются на стороне клиента алгоритмом **XChaCha20-Poly1305** (через libsodium). Сервер видит только зашифрованные данные и не может расшифровать содержимое файла.

### Формат ссылки для получателя

Отправитель генерирует ссылку, содержащую ID сессии и ключ шифрования в якоре URL (`#`). По стандарту HTTP якорь **никогда не отправляется на сервер** — он остаётся целиком в браузере.

```
https://example.com/#id=SESSION_ID&encryption=xchacha20-poly1305&key=BASE64URL_KEY
```

Якорь парсится как query-параметры:

| Параметр | Описание |
|---|---|
| `id` | ID сессии (совпадает с публичным ID создателя) |
| `encryption` | Идентификатор алгоритма шифрования. Сейчас: `xchacha20-poly1305` |
| `key` | Ключ шифрования в кодировке base64url (без паддинга) |

Поле `encryption` указывается явно, чтобы в будущем можно было добавить новые алгоритмы без нарушения совместимости со старыми ссылками.

### Схема шифрования

**Отправитель (для каждого чанка):**
1. Генерирует случайный 24-байтный nonce
2. Шифрует чанк: `XChaCha20-Poly1305(plaintext, nonce, key)` → ciphertext (включает 16-байтный тег аутентификации)
3. Присоединяет nonce к началу шифротекста: `[nonce (24 байта)][ciphertext]`
4. Отправляет результат на сервер как бинарный чанк

**Получатель (для каждого чанка):**
1. Получает бинарный чанк от сервера
2. Разделяет: первые 24 байта = nonce, остаток = ciphertext
3. Расшифровывает: `XChaCha20-Poly1305_Open(ciphertext, nonce, key)` → plaintext
4. Если расшифровка не удалась (неверный ключ или изменённые данные), чанк отбрасывается

**Генерация ключа:**
- Отправитель генерирует 256-битный (32-байтный) ключ функцией `crypto_aead_xchacha20poly1305_ietf_keygen()`
- Ключ кодируется в base64url (без паддинга) для включения в якорь URL

### Накладные расходы

Каждый зашифрованный чанк имеет фиксированный оверхед в **40 байт**: 24 байта nonce + 16 байт тег аутентификации Poly1305. При типичном размере чанка 5 МБ это пренебрежимо мало (~0.0008%).

# Общие эндпоинты

Простая часть логики, не связанная напрямую с передачей файлов.

### Текущие показатели нагрузки

При отрисовке интерфейса полезно отображать актуальные данные о текущей нагрузке сервера. Получение этих данных не требует сложных операций на стороне сервера, поэтому доступ к ним предоставляется без авторизации.

```
GET /api/statistics/current
```

Ответ всегда приходит с кодом `200`. Пример ответа:

```
{
  "max_session_count": 300,
  "max_user_count": 100,
  "current_session_count": 1,
  "current_user_count": 2
}
```

### Идентификационная информация

Позволяет получить информацию о текущей авторизации или узнать, что авторизация отсутствует. Авторизация позволяет участвовать только в одной сессии и сбрасывается при удалении сессии.

```
GET /api/me/info
```

Возможные ответы:

| Код | Тело | Значение |
|---|---|---|
| 401 | You have not been identified |  |
| 200 | **Описано ниже** | Сервер успешно распознал пользователя и вернул информацию о нём |

```
{
  "session": "",
  "name": "Data miner",
  "id": "2YrcZ1J9843a46tBnwF8UqqPkFEu5jEzuairz2myTBE"
}
```

Если поле `session` пустое, значит пользователь не присоединился ни к одной сессии.

### Сброс авторизации

```
POST /api/me/leave
```

Возможные ответы:

| Код | Тело | Значение |
|---|---|---|
| 401 | You have not been identified |  |
| 200 | Your ID has been deleted | Пользователь полностью удалён из системы |

# Создание идентификатора

Сервис не предусматривает привычную регистрацию с именем пользователя и паролем. Все пользователи остаются анонимными, но для борьбы с вредоносной нагрузкой и обеспечения работы логики приложения сервис устанавливает cookie.

Если нагрузка невелика, сервис установит cookie без активного участия пользователя. В противном случае будет предложено решить капчу.

В целях безопасности используется HttpOnly cookie, недоступная из контекста JS.

После создания пользователь будет удалён через минуту, если не установлено websocket-соединение.

### Запрос идентификации

```
GET /api/identity/request?name=Data miner
```

Для удобства взаимодействия пользователю необходимо указать произвольное имя (`name`). Оно будет отображаться другим участникам в рамках одной сессии передачи файлов.

Возможные ответы:

| Код | Тело | Значение |
|---|---|---|
| 400 | You are already authorized | Из cookie прочитан соответствующий идентификатор. |
| 503 | The maximum number of clients has been reached, please try again later | Достигнуто максимальное количество пользователей, установленное в конфигурационном файле. Запрос можно повторить через несколько секунд — это можно рассматривать как постановку нового пользователя в очередь. |
| 401 | **Описано ниже** | Необходимо решить капчу. |
| 201 | `{"name": "Applied name", "id": "Public ID"}` | Идентификатор выдан, в ответе также содержится заголовок для установки cookie. |

Если код ответа 401, необходимо показать пользователю капчу. Структура ответа сервера:

```
{
  "client_id": "Ef1rei8YGRDhdld2NNnSTEpG",
  "captcha_lifetime": 180,
  "captcha_token": "some loooong string",
  "captcha_image": "base64 encoded PNG",
  "captcha_answer_length": 5
}
```

`captcha_lifetime` — срок действия капчи в секундах. Его необходимо отобразить пользователю, например, чтобы он мог запросить новую капчу, если отвлёкся и не успел ответить.

### Подтверждение идентификации

```
POST /api/identity/confirmation

{
  "name": "Data miner",
  "client_id": "value-from-the-server-response",
  "captcha_token": "value-from-the-server-response",
  "captcha_answer": "user's input"
}
```

`name` — имя пользователя, указанное им при открытии интерфейса.

Возможные ответы:

| Код | Тело | Значение |
|---|---|---|
| 400 | *Текст ошибки* | Сообщает о некорректной сигнатуре запроса или о том, что в запросе уже обнаружена cookie с действующим идентификатором. |
| 403 | *Текст ошибки* | Сообщает, что ответ на капчу неверен, либо капча уже была использована и пользователь с данным кандидатным ID уже существует в системе. |
| 201 | `{"name": "Applied name", "id": "Public ID"}` | Идентификатор выдан, в ответе также содержится заголовок для установки cookie. |

## Подготовительная работа для передачи

В этом разделе описывается создание сессии и механизм присоединения к ней.

### Создание

```
POST /api/session/create

(пустое тело)
```

Возможные ответы:

| Код | Тело | Значение |
|---|---|---|
| 401 | You have not been identified | Cookie не найдена или содержит недействительный идентификатор. Необходимо начать процесс получения ID. |
| 403 | You are already a participant in the session | Пользователь уже является участником какой-либо сессии. |
| 503 | The maximum number of sessions has been reached, please try again later | Достигнуто максимальное количество сессий, указанное в конфигурационном файле. Запрос можно повторить через несколько секунд — это можно интерпретировать как постановку новой сессии в очередь. **Таймер автоматического удаления пользователя из-за неподключённого websocket сбрасывается при каждом таком запросе.** |
| 500 | Session creation failed | Скорее всего, произошла ошибка во внутренней логике сервера. |
| 201 | `{"id": "SESSION_ID"}` | Сессия создана. |

`SESSION_ID` использует безопасные для URL символы, поэтому его можно передавать в ссылке без риска нарушить логику.

В случае успеха необходимо установить websocket-соединение.

### Присоединение

```
GET /api/session/join?id=SESSION_ID
```

Возможные ответы:

| Код | Тело | Значение |
|---|---|---|
| 401 | You have not been identified | |
| 400 | *Текст ошибки*| Некорректный запрос |
| 404 | Session not found | Неверный ID или сессия была удалена |
| 403 | It is impossible to join the session | Сессия не может принять нового пользователя, так как достигнут лимит на количество получателей |
| 202 | `{"id": "SESSION_ID"}` | Присоединение к сессии прошло успешно |

В случае успеха необходимо установить websocket-соединение.

# WebSocket-соединение

```
/api/ws
```

Для подключения необходимо, чтобы cookie содержали действующий ID клиента и клиент был привязан к действующей сессии.
Допускается только одно активное соединение на пользователя (при переподключении предыдущее websocket-соединение будет закрыто).

Сразу после подключения сервер отправляет информацию, необходимую для инициализации состояния — событие `start_init`:

```
{
  "event": "start_init",
  "data": {
    "session_id": "FpXC3S1Mf8nV8hhBWzaQc9Fb3XEhFgKxaMJ9ZDDGXME",
    "transferred": {
      "global": {
        "to_receivers": 0,
        "from_sender": 0
      },
      "received_by_you": 0
    },
    "state": {
      "file": {
        "name": "",
        "size": 0
      },
      "expiration_in": 7200,
      "some_chunk_was_removed": false,
      "chunks": [
        {
          "index": 1,
          "size": 2356
        }
      ],
      "initial_freeze": true,
      "upload_finished": false,
      "current_chunk": 0
    },
    "members": {
      "sender": {
        "is_online": false,
        "name": "Data miner",
        "id": "FpXC3S1Mf8nV8hhBWzaQc9Fb3XEhFgKxaMJ9ZDDGXME"
      },
      "receivers": []
    },
    "limits": {
      "max_chunk_queue": 10,
      "max_initial_freeze": 120,
      "max_chunk_size": 5242880,
      "max_receiver_count": 5
    }
  }
}
```

Итоговый объект является общим для всех пользователей (для отправителя и получателей).

Если в `.state.file` имя пустое и размер равен нулю — это верный признак того, что информация о файле ещё не установлена.

**Подробное описание полей**:

- `session_id` — идентификатор сессии (совпадает с публичным ID создателя);
- `transferred` — объём переданных данных в байтах.
- `state` — глобальное состояние сессии
  - `expiration_in` — количество секунд до удаления сессии из-за достижения максимального времени жизни (таймаут);
  - `some_chunk_was_removed` — был ли удалён хотя бы один чанк. Если true, новые пользователи не могут присоединиться к такой сессии, так как часть данных потеряна;
  - `chunks` — информация о существующих чанках (при создании сессии чанков ещё нет, здесь приведён пример);
  - `initial_freeze` — начальная заморозка. Если активна, чанки не удаляются из буфера, даже если все известные получатели их скачали. Это позволяет новому получателю подключиться к сессии;
  - `upload_finished` — отправитель загрузил файл полностью;
  - `current_chunk` — номер последнего загруженного чанка.
- `members` — список участников
  - `sender` — отправитель. **Может быть равен `null`**, если пользователь отключился (но файл полностью загружен и сессия продолжает существовать);
  - `receivers` — массив других пользователей с аналогичными полями + `current_chunk`, который показывает номер последнего скачанного ими чанка.
- `limits` — глобальные лимиты
  - `max_chunk_queue` — максимальное количество чанков в буфере;
  - `max_initial_freeze` — максимальная продолжительность начальной заморозки в секундах (при достижении заморозка сбрасывается автоматически). **Этот параметр также определяет максимальное время ожидания получателей. Если их нет, сессия будет завершена. Если информация о файле не установлена на момент сброса начальной заморозки, сессия также будет уничтожена**;
  - `max_chunk_size` — максимальный размер чанка, который может загрузить отправитель;
  - `max_receiver_count` — максимальное количество получателей.

---

Все текстовые сообщения websocket содержат JSON. Сообщения сервера необходимо обрабатывать по полю `event`, а все входящие сообщения от клиента должны содержать ключ `action`. Полезная нагрузка всегда должна находиться в объекте `data`. Если сообщение грубо нарушает ожидаемый формат, то есть нарушает описанный здесь протокол, соединение закрывается с текстовым описанием проблемы и кодом 1003 (Unsupported Data).

## События, отправляемые сервером

### E1. Связанные с другими пользователями

#### E1.1 Онлайн

Уведомляет об активном websocket-соединении. Если пользователь не подключён длительное время (60 секунд), он удаляется.

```
{
  "event": "online",
  "data": {
    "id": "user's id",
    "status": true
  }
}
```

#### E1.2 Смена имени

```
{
  "event": "name_changed",
  "data": {
    "id": "user's id",
    "name": "New user's name"
  }
}
```

### E2. Связанные с сессией передачи

#### E2.1 Новый получатель в сессии

```
{
  "event": "new_receiver",
  "data": {
    "id": "user's id",
    "name": "Username"
  }
}
```

#### E2.2 Клиент удалён

```
{
  "event": "receiver_removed",
  "data": {
    "id": "user's id"
  }
}
```

#### E2.3 Информация о файле установлена

```
{
  "event": "file_info",
  "data": {
    "name": "archive.zip",
    "size": 61359353
  }
}
```

Размер указан в байтах.

#### E2.4 Доступен новый чанк

Загружен новый чанк, доступный для скачивания.

```
{
  "event": "new_chunk",
  "data": {
    "index": 31,
    "size": 1536282
  }
}
```

Размер указан в байтах.

#### E2.5 Скачивание чанка

Событие указывает на начало скачивания и завершение, когда получатель явно подтверждает приём.

```
{
  "event": "chunk_download",
  "data": {
    "id": "user's id",
    "index": 31,
    "action": "started" or "finished"
  }
}
```

#### E2.6 Чанк удалён

```
{
  "event": "chunk_removed",
  "data": {
    "id": [
      78
    ]
  }
}
```

#### E2.7 Счётчик общего объёма переданных данных увеличился

```
{
  "event": "bytes_count",
  "data": {
    "value": 3927538,
    "direction": "from_sender" or "to_receivers"
  }
}
```

#### E2.8 Объём полученных данных

```
{
  "event": "personal_received",
  "data": {
    "bytes": 329273538
  }
}
```

#### E2.9 Начальная заморозка чанков сброшена

```
{
  "event": "chunks_unfrozen",
  "data": {}
}
```

#### E2.10 Загрузка файла завершена

После этого события новые чанки не могут быть созданы.

```
{
  "event": "upload_finished",
  "data": {}
}
```

#### E2.11 Сессия завершена

```
{
  "event": "complete",
  "data": {
    "status": "Read below"
  }
}
```

Возможные статусы:
- `ok` — нормальное завершение;
- `timeout` — сессия удалена по таймауту;
- `sender_is_gone` — отправитель покинул сессию;
- `no_receivers` — получатели отсутствуют (или так и не подключились);
- `error` — другая ошибка.

#### E2.12 Доступна возможность загрузки нового чанка

Событие предназначено только для создателя сессии. Уведомляет о том, что загрузка нового чанка разрешена. Если не разрешена — буфер заполнен.

```
{
  "event": "new_chunk_allowed",
  "data": {
    "status": false
  }
}
```

## Действия, отправляемые пользователем

### A1. Административные действия создателя сессии

#### A1.1 Установка информации о файле

Может быть указана только один раз. Размер указывается в байтах.

```
{
  "action": "set_file_info",
  "data": {
    "name": "my_video.mp4",
    "size": 17654927
  }
}
```

Возможна ошибка, если информация уже была установлена, или размер равен нулю, или передано пустое имя, или длина имени превышает лимит (255 байт). В случае ошибки возвращается индивидуальное событие:

```
{
  "event": "set_file_info_failure",
  "data": {}
}
```

#### A1.2 Загрузка файла завершена

```
{
  "action": "upload_finished",
  "data": {}
}
```

#### A1.3 Удаление получателя

```
{
  "action": "kick_receiver",
  "data": {
    "id": "another-user-id"
  }
}
```

Создатель сессии не может удалить самого себя — это приведёт к ошибке.

#### A1.4 Принудительное завершение сессии

```
{
  "action": "terminate_session",
  "data": {}
}
```

#### A1.5 Снятие начальной заморозки

Вручную снимает начальную заморозку чанков. Пока заморозка активна, чанки не удаляются из буфера даже после того, как все текущие получатели их скачали. Это позволяет новым получателям подключиться и получить начальные данные. После снятия заморозки (вручную или по таймеру) чанки начнут освобождаться после подтверждения, позволяя передаче продолжиться.

```
{
  "action": "drop_freeze",
  "data": {}
}
```

Если заморозка уже снята, действие игнорируется.

#### A1.6 Отправка нового чанка

Создатель сессии может отправлять бинарные данные. Они будут обработаны как новый чанк и добавлены в буфер сессии. Если добавление в буфер не удалось, возвращается индивидуальное событие:

```
{
  "event": "add_chunk_failure",
  "data": {}
}
```

Ошибка возможна, если буфер чанков заполнен.

### A2. Общие действия

#### A2.1 Смена имени

```
{
  "action": "new_name",
  "data": {
    "name": "The lazy snail"
  }
}
```

Имя должно быть в формате UTF-8. Длина имени автоматически сокращается до максимально допустимой (20 символов).

#### A2.2 Получение чанка

```
{
  "action": "get_chunk",
  "data": {
    "index": 78
  }
}
```

Если запрошенный фрагмент не найден, возвращается отдельное событие с информацией о существующих чанках:

```
{
  "event": "requested_chunk_not_found",
  "data": {
    "available": [
      {
        "index": 79,
        "size": 62857
      },
      {
        "index": 80,
        "size": 62857
      }
    ]
  }
}
```

В случае успеха ответ содержит бинарные данные запрошенного чанка.

#### A2.3 Подтверждение получения чанка

Чанк считается полученным только после явного подтверждения. Когда чанк получен всеми получателями, он удаляется из буфера, освобождая место для нового чанка.

```
{
  "action": "confirm_chunk",
  "data": {
    "index": 81
  }
}
```
